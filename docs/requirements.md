## 1. システム概要

### 1.1 目的

療養中の家族を含む健康管理を考慮しつつ、栄養バランスの取れた献立を管理・提案するシステム。

料理が苦手なユーザーでも簡単に使え、かつ飽きの来ない献立レパートリーを構築することを目的とする。

### 1.2 ターゲットユーザー

- **主なユーザー:** 料理が苦手だが、家族の健康管理のために献立を考えたい人
- **対象家族:** 療養中の家族（塩分・脂質制限が必要だが、完全制限は避けたい）

### 1.3 基本ポリシー

- **制限すぎず、バランス重視:** たまにはとんかつもOK（しょっちゅうはダメ）
- **栄養考慮:** タンパク質、塩分、カロリーを意識
- **簡単入力:** 料理苦手な人でもレシピ登録しやすい
- **逆引き検索:** 「キャベツとサバ缶で何作れる？」といった検索に対応

## 2. 機能要件

### 2.1 コア機能

#### 2.1.1 献立・レシピ管理

**献立データ:**
- 5種類の献立タイプ（主菜、副菜、汁物、主食、デザート）
- 各献立の基本情報（名前、説明、難易度、調理時間、人数）
- 栄養情報（総量: タンパク質量、カロリー、塩分、脂質）
- 血栓の病気に良いかどうかのフラグ
- タグ管理（フリーフォーマット入力）

**レシピデータ:**
- 材料と分量（正規化された食材マスターと紐付け）
- 調理手順（ステップバイステップ）
- 調理のヒント・コツ

#### 2.1.2 CRUD操作（Web UI）

**すべてのデータに対して:**
- **追加:** Web UIから献立・レシピ・食材を追加
- **変更:** 既存データを編集
- **削除:** 不要なデータを削除
- **参照:** 一覧表示、詳細表示

#### 2.1.3 OpenClawスキル経由の操作

**可能な操作:**
- **追加:** 献立・レシピ・食材を登録
- **変更:** 既存データを更新
- **参照:** 検索、一覧取得、詳細取得

**禁止する操作:**
- **削除:** スキルからの削除は禁止（Web UIのみ）

#### 2.1.4 食材検索・逆引き

**ChromaDBによるセマンティック検索:**
- 食材から献立を検索（「キャベツと卵で作れるもの」）
- あいまい検索（「余ってる野菜で何作れる？」）
- 類似献立の推薦

**データ連動:**
- 献立・レシピが変更されたら、ChromaDBも自動更新
- 食材が変更されたら、関連するChromaDBを更新

#### 2.1.5 献立提案

**AIによる提案:**
- 主菜+副菜2+汁物のバランスを考慮
- 栄養バランスチェック（塩分合計、タンパク質合計など）
- カレンダー機能（週間献立カレンダー）

**提案条件:**
- ユーザーの指定があれば優先（「魚料理がいい」など）
- 好き嫌いの考慮（将来的な拡張）
- 季節の食材の考慮（将来的な拡張）

#### 2.1.6 カレンダー機能

**1日3食カレンダー:**
- 1日の献立を3食（朝食・昼食・夕食）で管理
- 日付ごとの献立（主菜、副菜1、副菜2、汁物、主食、デザート）
- カレンダーから献立を選んで詳細表示
- 1日分の総栄養表示（カロリー、タンパク質、塩分）
- **人数考慮:** レシピの人数で栄養情報を1人前に換算

### 2.3 OpenClawスキル要件

#### 2.3.1 提供される機能

**献立登録:**
```
kondate add dish --name "サバ缶ツナサラダ" \
  --type "side" \
  --recipe "キャベツ1/4玉、サバ水煮缶1/2、マヨネーズ大さじ1" \
  --steps "キャベツを千切りにする、サバ缶をほぐして混ぜる、マヨネーズをかける"
```

**献立検索:**
```
kondate search dish "サバ"
kondate search ingredients "キャベツ 卵"
```

**献立提案:**
```
kondate suggest --type "dinner" --prefer "fish"
kondate suggest calendar --week "2025-01-21"
```

**データ更新:**
```
kondate update dish --id 1 --calories 250
kondate update dish --id 1 --add-ingredient "卵"
```

#### 2.3.2 スキルインターフェース

**Pythonクライアント:**
- `skills/kondate/scripts/client.py`
- uvで管理されたPythonスクリプト
- APIエンドポイントを呼び出すラッパー

**シェルスクリプトラッパー（将来的な拡張）:**
- 簡単なコマンド用

## 3. 非機能要件

### 3.1 パフォーマンス

- **検索応答時間:** ChromaDB検索は1秒以内
- **UI応答時間:** ページ読み込みは2秒以内
- **同時アクセス:** 家族内共有レベル（数名同時アクセス）

### 3.2 セキュリティ

- **認証:** ローカル運用前提のため認証なし
- **データ保護:** ボリュームマウントで永続化
- **インジェクション対策:** SQLインジェクション、XSS対策

### 3.3 可用性

- **バックアップ:** SQLiteデータの定期的バックアップ（ユーザー運用）
- **Docker:** コンテナ化で環境差異を吸収
- **ログ:** エラーログの記録

### 3.4 保守性

- **ドキュメント:** 導入マニュアル、APIドキュメント（Swagger UI）
- **コード品質:** PEP8準拠、型ヒント使用
- **GitHub公開:** オープンソースとして公開

## 4. データ要件

### 4.1 データ項目

#### 4.1.1 献立（Dishes）

| 項目 | 型 | 必須 | 説明 |
|------|------|------|------|
| id | Integer | ✓ | 主キー |
| name | String | ✓ | 献立名 |
| type | Enum | ✓ | main（主菜）/side（副菜）/soup（汁物）/staple（主食）/dessert（デザート） |
| description | Text | ✓ | 説明 |
| difficulty | Enum | ✓ | easy/medium/hard |
| prep_time | Integer | ✓ | 調理時間（分） |
| servings | Integer | ✓ | 人数（デフォルト2） |
| calories | Float | ✓ | カロリー（kcal/総量） |
| protein | Float | ✓ | タンパク質（g/総量） |
| fat | Float | ✓ | 脂質（g/総量） |
| sodium | Float | ✓ | 塩分（g/総量） |
| good_for_brain_health | Boolean | | 血栓の病気に良いかどうか |
| created_at | DateTime | ✓ | 作成日時 |
| updated_at | DateTime | ✓ | 更新日時 |

#### 4.1.2 レシピ（Recipes）

| 項目 | 型 | 必須 | 説明 |
|------|------|------|------|
| id | Integer | ✓ | 主キー |
| dish_id | Integer | ✓ | 外部キー（dishes.id） |
| ingredients | Text | ✓ | 材料（テキスト形式） |
| steps | JSON | ✓ | 調理手順 |
| tips | Text | ✓ | コツ・ヒント |

#### 4.1.3 食材（Ingredients）

| 項目 | 型 | 必須 | 説明 |
|------|------|------|------|
| id | Integer | ✓ | 主キー |
| name | String | ✓ | 食材名 |
| category | String | | カテゴリ（野菜、肉、魚など） |
| season | String | | 旬の季節 |

#### 4.1.4 献立-食材関連（DishIngredients）

| 項目 | 型 | 必須 | 説明 |
|------|------|------|------|
| id | Integer | ✓ | 主キー |
| dish_id | Integer | ✓ | 外部キー（dishes.id） |
| ingredient_id | Integer | ✓ | 外部キー（ingredients.id） |
| amount | String | ✓ | 分量（例: "1/2個", "100g"） |

#### 4.1.5 タグ（Tags）

| 項目 | 型 | 必須 | 説明 |
|------|------|------|------|
| id | Integer | ✓ | 主キー |
| name | String | ✓ | タグ名 |

#### 4.1.6 献立-タグ関連（DishTags）

| 項目 | 型 | 必須 | 説明 |
|------|------|------|------|
| id | Integer | ✓ | 主キー |
| dish_id | Integer | ✓ | 外部キー（dishes.id） |
| tag_id | Integer | ✓ | 外部キー（tags.id） |

#### 4.1.7 カレンダー（Calendar）

| 項目 | 型 | 必須 | 説明 |
|------|------|------|------|
| id | Integer | ✓ | 主キー |
| date | Date | ✓ | 日付 |
| meal_type | String | ✓ | 食事タイプ（breakfast/lunch/dinner） |
| main_dish_id | Integer | | 外部キー（dishes.id） |
| side1_dish_id | Integer | | 外部キー（dishes.id） |
| side2_dish_id | Integer | | 外部キー（dishes.id） |
| soup_dish_id | Integer | | 外部キー（dishes.id） |
| staple_dish_id | Integer | | 外部キー（dishes.id） |
| dessert_dish_id | Integer | | 外部キー（dishes.id） |

**制約:**
- `date + meal_type` の組み合わせは一意

### 4.2 ChromaDB設計

#### 4.2.1 コレクション: dishes

**ドキュメント構造:**
```json
{
  "id": 1,
  "name": "サバ缶ツナサラダ",
  "type": "side",
  "description": "キャベツとサバ缶で簡単サラダ",
  "ingredients_summary": "キャベツ, サバの水煮缶, マヨネーズ",
  "tags": ["魚介類", "簡単", "脳梗塞に良い"],
  "good_for_brain_health": true
}
```

**メタデータ:**
```json
{
  "dish_id": 1,
  "type": "side",
  "calories": 180,
  "protein": 12,
  "sodium": 0.8,
  "good_for_blood_health": true
}
```

**埋め込み用途:**
- 食材からの逆引き（「キャベツ 卵」→ 類似献立検索）
- 説明文からのセマンティック検索
- タグからの検索

## 5. API要件

### 5.1 REST APIエンドポイント

#### 5.1.1 献立（Dishes）

| メソッド | エンドポイント | 説明 |
|---------|---------------|------|
| GET | /api/dishes | 献立一覧取得 |
| GET | /api/dishes/{id} | 献立詳細取得 |
| POST | /api/dishes | 献立登録 |
| PUT | /api/dishes/{id} | 献立更新 |
| DELETE | /api/dishes/{id} | 献立削除 |

#### 5.1.2 検索

| メソッド | エンドポイント | 説明 |
|---------|---------------|------|
| GET | /api/search/dishes?q={query} | 献立名検索 |
| GET | /api/search/ingredients?q={query} | 食材から献立検索（ChromaDB） |
| GET | /api/search/similar?id={id} | 類似献立検索 |

#### 5.1.3 提案

| メソッド | エンドポイント | 説明 |
|---------|---------------|------|
| GET | /api/suggest?prefer={type} | 献立提案 |
| GET | /api/suggest/calendar?week={date} | 週間献立カレンダー提案 |

#### 5.1.4 カレンダー

| メソッド | エンドポイント | 説明 |
|---------|---------------|------|
| GET | /api/calendar?start={date}&end={date} | カレンダー取得（日付範囲） |
| GET | /api/calendar?year={year}&month={month} | カレンダー取得（月間） |
| GET | /api/calendar?date={date} | 1日分の3食を取得 |
| GET | /api/calendar/{date} | 日付でカレンダーを取得（3食） |
| POST | /api/calendar | カレンダー登録（既存の場合409 Conflict） |
| PUT | /api/calendar/{date}?meal_type={type} | カレンダー更新 |
| DELETE | /api/calendar/{date}?meal_type={type} | カレンダー削除 |

**エラーハンドリング:**
- **409 Conflict:** 既存登録時にPOSTを試みた場合
  - エラーメッセージ: "{date}の{meal_type}は既に登録されています。更新するにはPUTメソッドを使用してください。"
  - クライアントはユーザー確認後、PUTで更新

#### 5.1.5 食材・タグ

| メソッド | エンドポイント | 説明 |
|---------|---------------|------|
| GET | /api/ingredients | 食材一覧 |
| POST | /api/ingredients | 食材登録 |
| GET | /api/tags | タグ一覧 |
| POST | /api/tags | タグ登録 |

## 6. 技術要件

### 6.1 バックエンド

- **言語:** Python 3.11+
- **フレームワーク:** FastAPI
- **ORM:** SQLAlchemy
- **データベース:** SQLite3
- **ベクトルDB:** ChromaDB
- **Webサーバー:** Uvicorn

### 6.2 フロントエンド

- **テンプレートエンジン:** Jinja2
- **CSS:** 独自CSS（シンプル・レスポンシブ）
- **JavaScript:** Vanilla JavaScript（将来的な拡張でフレームワーク検討）

### 6.3 インフラ

- **コンテナ:** Docker
- **オーケストレーション:** Docker Compose
- **パッケージ管理:** uv
- **ボリュームマウント:**
  - `src/datas` → データ永続化
  - `src/models` → 将来的なMLモデル拡張用

### 6.4 開発ツール

- **バージョン管理:** Git + GitHub
- **APIドキュメント:** Swagger UI（FastAPI自動生成）
- **テスト:** pytest（将来的な拡張）

## 7. 制約事項

### 7.1 技術的制約

- **認証なし:** ローカル運用前提
- **同時アクセス:** 数名レベル（スケーラビリティは求めない）
- **データバックアップ:** ユーザー責任（SQLiteファイルのコピー）

### 7.2 機能的制約（初版）

- **ユーザー管理:** なし（将来的な拡張）
- **画像管理:** なし（将来的な拡張）
- **栄養計算自動化:** 手動入力（将来的な拡張でAPI連携検討）
- **AI提案:** シンプルなランダム・ルールベース（将来的な拡張でLLM連携検討）
